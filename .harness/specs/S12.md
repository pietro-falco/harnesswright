# S12 — ADR-004 implementation: spec validator, Mode B eligibility, lock read

First implementation slice against ADR-003 (Accepted 2026-07-10) and ADR-004
(Accepted 2026-07-11). It makes two of ADR-004's contracts executable and
leaves the rest as docs:

1. **Spec validator** (`src/spec.ts`) — a restricted-YAML frontmatter parser in
   the style of `parseHarnessConfig`: scalars, flat lists, one-level maps, no
   runtime dependency. Fields and constraints are ADR-003's set plus ADR-004
   D1's delta (`status`, `scope`, `model`) and its budget units. `gate-failure`
   is unioned into `stop_conditions` and has no negation syntax. An unknown key
   or an invalid value is a configuration error — exit-2 discipline.
2. **Eligibility in `next --json`** (ADR-004 D2) — for the unlocked slice,
   `eligible_mode_b` is a total predicate: an accepted, mode-B spec exists and
   no lock is present at `.harness/locks/<id>.lock`. The lock is read for
   presence only; its content is audit-only and never parsed. `next` stays
   read-only with the 0/2 exit vocabulary unchanged, and the effective model is
   reported as a tier (`worker` | `executor`, D8) unless declared.

Out of scope, by the operator's directive: co-executability and
`workstreamCap > 1` (D4), the Stop hook and closure receipts (D7),
notifications, scheduling, version bump, publish.

**Evidence:** `node --test` green (110 tests), `tsc --noEmit` exit 0, and this
spec validating against the validator it ships.
